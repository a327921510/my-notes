## 0. 工程版本与文档迭代约定

### 0.1 版本号与文档的对应关系

- **单一来源**：仓库根目录 `package.json` 的 `version` 字段为**工程语义化版本号**（如 `0.1.0`）。
- **清单**：`docs/version-manifest.json` 记录当前版本、上一版本（若有）及 `docs/` 下各文件与版本的关联；迭代文档或发版时请一并更新该文件。
- **按版本快照**：`docs/versions/` 用于存放已发布版本的文档快照，流程说明见 [versions/README.md](./versions/README.md)。

### 0.2 版本迭代时如何查阅文档

每次做**版本迭代**（需求评审、架构调整、实现与联调、发版说明）时，应**同时对照**以下两类材料，避免只改当前文档而忽略与上一版的差异：

1. **当前版本**：`docs/` 目录下现行文档（及 `version-manifest.json` 中 `currentVersion`）。
2. **上一版本**：`version-manifest.json` 中的 `previousVersion` 非空时，以 `docs/versions/<previousVersion>/` 中的快照为准；若无快照，则以 Git 中该版本标签或分支上的 `docs/` 为准。

首个基线版本为 `0.1.0`，此时无上一版快照属正常情况；从下一版起在升版前按 `versions/README.md` 落盘快照即可形成可对照的「上一版本」。

### 0.3 面向 AI 与自动化工具的定位（推荐）

若主要目标是**让 AI 在开发时更快比对两版需求差异**，建议采用「三层」存放，各司其职：

1. `**docs/version-manifest.json`**：版本号与文件列表的单一事实来源，便于脚本或 AI 先定位要比对哪些路径。
2. `**docs/versions/<版本>/` 快照**：与现行 `docs/` 同结构的**静态副本**，便于做逐文件 diff（比纯靠记忆或口头描述更稳）。
3. `**.cursor/rules/docs-version-diff.mdc`**：在 Cursor 中写入**固定比对步骤**（默认注入），减少漏比、漏读上一版。

仓库根目录 `**AGENTS.md`** 对上述路径有一页索引，供非 Cursor 的代理或工具链引用。

### 0.4 需求落地与版本升级（补充约定）

- **文档为最终归口**：无论需求来自对话、工单还是口头，**凡形成或变更的需求**，最终都应**整理、汇总进 `docs/`**（至少更新 `PRD.md` 或 `开发文档.md` 中对应小节；涉及流程/结构时同步 `用户流程图.md`、`产品功能结构图.md` 等）。实现与评审以仓库内文档为准，避免「只改代码、文档滞后」。
- **识别是否升版**：在梳理或评审需求时，应主动识别两类信息：
  - **是否明确要求升版**：需求方或 PRD 是否写明「发版」「版本号」「里程碑」等与工程版本挂钩的表述；若有，落实时需同步更新根目录 `package.json` 的 `version`、`docs/version-manifest.json`，并按 [versions/README.md](./versions/README.md) 处理快照。
  - **是否建议升版**：即使未写明版本号，若变更属于**对外行为、数据格式、接口契约、导出格式、兼容性或验收标准**的实质变化，应**提示是否有必要**进行语义化版本升级（通常至少 PATCH；破坏性变更需 MINOR/MAJOR，按团队 semver 约定执行），避免静默改行为。

## 1. 仓库与工程结构

### 1.1 Monorepo（npm workspaces）

根目录 `package.json` 定义 workspaces：`packages/`*、`apps/*`。


| 路径                | 包名                 | 职责                                                       |
| ----------------- | ------------------ | -------------------------------------------------------- |
| `packages/shared` | `@my-notes/shared` | 跨端共享类型与常量；**纯函数**：`createId`、`nextSyncAfterEdit`、`needsUpload`、云端拉取合并判定等 |
| `packages/local-db` | `@my-notes/local-db` | Web 与扩展共用的 Dexie 定义（`NotesDB`、单例 `db`）及 `seed`（默认文件夹等） |
| `packages/sync-client` | `@my-notes/sync-client` | Web 与扩展共用的**副作用型**同步：`pullFromCloud`、`uploadNote`、`uploadSnippet`（`fetch` + 写本地库） |
| `apps/web`        | `@my-notes/web`    | Web 主应用：Vite + React + React Router，本地库来自 `@my-notes/local-db` |
| `apps/api`        | `@my-notes/api`    | 轻量后端：Fastify，JWT 鉴权、图片上传、笔记/短文本 upsert（内存存储，MVP）         |
| `apps/extension`  | `@my-notes/extension` | 浏览器扩展：短文本 UI；IndexedDB 与 Web 同 schema；登录与手动同步走 `sync-client` |


常用脚本（根目录）：

- `npm run dev`：启动 Web 开发服务（`@my-notes/web`）。
- `npm run dev:api`：启动 API（默认端口见 `apps/api`）。
- `npm run build`：构建 shared、local-db、sync-client、web、api、extension。

**约定**：Node 版本 `>= 20`（见根 `package.json` `engines`）。

## 2. 技术栈与框架设计

### 2.1 Web（`apps/web`）

- **构建**：Vite 6 + TypeScript。
- **UI**：React 19；**页面与交互组件使用 Ant Design（`antd`）** 搭建（布局、表单、数据展示、导航、反馈等）。当前依赖：`antd` ^6.x、`@ant-design/icons` ^6.x（见 `apps/web/package.json`）。入口 `main.tsx` 使用 `ConfigProvider` + `locale={zh_CN}`，并以 `<App>` 包裹路由以启用 `App.useApp()`（`message` 等）。样式层可保留 **Tailwind CSS 4**（`@tailwindcss/vite`）作少量工具类或布局补充；与 `antd` 混用时避免在同一元素上堆叠冲突的样式，优先以组件 API / `classNames` / `style` 与 Design Token 为准。
- **路由**：`react-router-dom` v7，`App.tsx` 中定义顶层路由；`Layout` 提供 Tab 导航（笔记区 / 站点信息区 / 上传中心 / 分层范例等）。
- **本地数据**：Dexie 封装见 **`@my-notes/local-db`**（`NotesDB`，库名 `my_notes_v1`）；表含 `folders`、`notes`、`images`、`snippets`、`site_spaces`、`clips`、`blobs`、`sites`、`site_items`、`drive_folders`、`drive_files`、`drive_file_tombstones`。云盘文件二进制存 `blobs`，`drive_file_tombstones` 用于本地删除后补偿上行删除云端。
- **共享类型与纯同步语义**：业务实体与合并判定从 `@my-notes/shared` 引入；跨端拉取/上传副作用从 `@my-notes/sync-client` 引入。

### 2.2 API（`apps/api`）

- **框架**：Fastify 5。
- **插件**：`@fastify/cors`、`@fastify/jwt`、`@fastify/multipart`（笔记推送含多图二进制、单图上传）。
- **鉴权**：`/api/auth/login` 签发 JWT；`/api/upload/*`、`/api/notes/*`、`/api/snippets/*`、`/api/sites/*`、`/api/site-items/*`、`/api/drive/*`、`/api/files/*` 需 JWT（`preHandler` 中校验）。
- **笔记推送**：`POST /api/notes/push`（`multipart`：`meta` JSON + `img_<clientImageId>` 文件流），服务端落盘并返回 `storageId`；**不再**通过公开 URL 暴露上传文件。
- **文件拉取**：`GET /api/files/:storageId` 返回文件流（需 JWT，与所属用户绑定）。
- **云盘文件**：
  - `POST /api/drive/folders/upsert`、`GET /api/drive/folders`；
  - `POST /api/drive/files/push`（`multipart`：`meta` + `file` 二进制）；
  - `GET /api/drive/files`、`GET /api/drive/files/:cloudId/download`；
  - `DELETE /api/drive/files/:clientFileId`（供 tombstone 同步删除云端）。
- **删除云端**：`DELETE /api/notes/:clientNoteId`、`DELETE /api/snippets/:clientSnippetId`；站点域与短文本对称：`DELETE /api/sites/:clientSiteId`（级联删除该站下云端条目）、`DELETE /api/site-items/:clientItemId`。
- **现状**：笔记与短文本云端存储为进程内 `Map`，重启丢失；生产需替换为持久化存储。已提供 `GET /api/notes`、`GET /api/snippets` 供 Web 端手动拉取列表并合并。

### 2.3 共享包（`packages/shared`）

- 导出 `SyncStatus`、`NoteRecord`、`ImageRecord`、`SnippetRecord` 等，与 PRD 第 8 节逻辑模型一致。
- `SYNC_STATUS_LABELS`、`SNIPPET_TYPE_LABELS` 供 UI 展示，避免魔法字符串散落。

### 2.4 扩展（`apps/extension`）

- **Vite 构建**；依赖 `@my-notes/shared`、`@my-notes/local-db`、`@my-notes/sync-client`、`dexie`。
- **API 基址**：扩展内不能使用站点相对路径访问后端，需配置 **`import.meta.env.VITE_API_BASE`**（如开发环境 `http://127.0.0.1:3001`，与 Web 开发代理目标一致），构建时由 Vite `define` 注入；`sync-client` 将 `${VITE_API_BASE}/api/...` 拼为绝对 URL。
- **Manifest**：对 API 源配置 **`host_permissions`**（或等价权限），否则 `fetch` 会被浏览器拦截。
- **会话**：建议将 JWT 与用户摘要存入 **`chrome.storage.session`**（扩展进程结束时清除；与 Web 使用 `localStorage` 不同，文档与实现需一致）。
- **迁移**：旧版 `chrome.storage.local` 中的短文本列表可一次性导入 `IndexedDB` 后清除，避免双写。

## 3. 开发约定

### 3.1 语言与风格

- **TypeScript**：严格模式；公共实体类型以 `shared` 为单一来源。
- **路径别名**：Web 端使用 `@/` 指向 `src/`（见 `vite.config.ts` / `tsconfig`）。
- **组件**：页面级放在 `src/pages/`，布局与导航在 `src/components/`。

### 3.1.1 UI：Ant Design 约定

- **选型**：Web 端统一采用 **Ant Design**（npm 包名 `antd`）作为 UI 组件库；新页面、新功能**优先使用 antd 提供的组件**（如 `Layout`、`Menu`、`Table`、`Form`、`Input`、`Button`、`Modal`、`message`、`notification`、`Upload`、`Tabs` 等），减少手写无样式结构 + 零散 CSS。
- **全局配置**：在应用入口（如 `main.tsx` 或根组件）使用 `ConfigProvider` 统一主题与**中文语言包**（`import zhCN from 'antd/locale/zh_CN'`），保证文案与日期等本地化一致。
- **与 Tailwind 的关系**：历史或局部仍可有 Tailwind 类名；**新增界面以 antd 为主**。若需深度定制外观，优先使用 antd 5 的 **Design Token**（`theme`）与 `ConfigProvider`，避免大量 `!important` 覆盖组件内部样式。
- **依赖**：`apps/web` 已安装 `antd`、`@ant-design/icons`；版本见 `package.json`。

### 3.1.2 页面分层约定（入口 / 区域组件 / Hook / 纯展示）

- **目标**：在「单文件页面」可演进的前提下，用 **页面入口、区域组件、业务 Hooks、纯展示组件** 四块划分职责；降低耦合并减少无意义透传。
- **主规则**（细则与检查清单见 `.cursor/rules/page-layering.mdc`）：
  1. **页面入口**（`src/pages/<PageName>/index.tsx`）：串联各区块；跨组件数据、通讯与联动在此编排。
  2. **区域组件**：相对独立的一整块 UI（如左列表面板）；对内自洽，对外只约定少量共享状态或回调（如 `onSelectionChange`）；**可**调用本页 `hooks/` 以复用数据操作契约。
  3. **业务 Hooks**（`src/pages/<PageName>/hooks/use*.ts`）：与具体布局无关的存取与变更；对外 `state + actions`。
  4. **纯展示组件**：只根据 props 展示并发事件；不直连 Dexie/HTTP/本地存储，不承担复杂跨区编排。
- **范例（仓库内）**：`apps/web/src/pages/PageLayeringDemo/`，路由 `/layering-demo`。文件与职责对应如下：

| 路径 | 角色 |
|------|------|
| `PageLayeringDemo/index.tsx` | 页面入口 |
| `PageLayeringDemo/components/DemoListPanel.tsx` | 区域组件 |
| `PageLayeringDemo/hooks/useDemoItems.ts` | 业务 Hook |
| `PageLayeringDemo/components/DemoStatBar.tsx`、`DemoDetailView.tsx` | 纯展示 |

- **目录建议（Web）**：页面入口放在 `src/pages/<PageName>/index.tsx`；区域与纯展示放在 `src/pages/<PageName>/components/` 或 `src/components/`；Hook 放在 `src/pages/<PageName>/hooks/` 或 `src/hooks/`。
- **拆分策略**：最小侵入，优先抽出高复用或高复杂度区块。
- **补充（旧「三层」表述）**：原「展示 / Hook / 页面控制器」叙述仍可作为理解入口与 Hook 的别名；**纯展示**与**区域组件**的区分以 `.cursor/rules/page-layering.mdc` 为准（区域组件可调用本页 Hook；纯展示不调数据层）。
- **边界检查**：是否存在「纯展示直连数据库」「跨区联动散落在区域深处」「为少传 props 引入隐式全局耦合」等反模式。

### 3.2 数据与同步语义

- **同步状态**（`SyncStatus`）：`local_only` | `synced` | `dirty` | `failed`，含义见 PRD §5。
- **图片**：`ImageRecord.localBlobRef` 对应 `blobs` 表中的 `key`；推送时 `uploadNote` 使用 `multipart` 将二进制一并提交；同步成功后 `cloudStorageId` 存服务端 `storageId`。拉取时用 JWT 请求 `/api/files/:storageId` 得到 `Blob` 再写入本地 `blobs`。
- **正文**：笔记 `contentText` 存 **HTML**（富文本编辑器输出），搜索与标题摘要前用 `stripHtml` 做纯文本处理（见 `src/lib/html.ts`）。
- **推送**：`@my-notes/sync-client` 中 `uploadNote` / `uploadSnippet`；**是否需要上传**由 `@my-notes/shared` 的 `needsUpload` 判定。
- **拉取合并（纯函数）**：笔记/短文本与云端比较、冲突副本标题等逻辑在 `@my-notes/shared`（如 `hasNoteConflict`、`decideNotePull`、`snippetPullShouldApply`、`buildSnippetRowFromCloudPull`）；Dexie 写入与网络请求在 `@my-notes/sync-client`。
- **短文本（站点信息区 / 扩展）下行合并**：云端覆盖 `type`、正文、域名、`sourceUrl`、`updatedAt`、`syncStatus`、`cloudId` 等同步载荷字段；**本地 IndexedDB 已有、且当前 API 未返回的字段**（如 `sourceTitle`、已存在的 `createdAt`）在拉取时**暂保留**，避免扩展采集的标题等被空数据冲掉。
- **登录态同步规则**：
  - 未登录：本地与云端隔离，不触发自动同步；
  - 登录成功：自动执行一次 `pullFromCloud`；
  - 云端有本地无：直接落本地；
  - 本地有云端无：保持本地（`local_only`）；
  - 云端本地都有且内容不一致：同时保留两份（新增云端副本）。

### 4.1.5 交互与状态（示例）

- Hover：卡片提升一层阴影并显示快捷操作（置顶、删除、更多）。
- 选中：高亮边框 + 背景弱色，右侧预览同步切换。
- 空状态：显示引导文案与“新建笔记”主按钮。
- 加载状态：列表骨架屏；同步中显示 `SyncBadge` 状态提示。

### 4.1.6 验收标准（占位）

- 在常见桌面分辨率下（1366x768、1920x1080）无需横向滚动即可完成核心操作。
- 新用户可在 10 秒内完成“新建并保存一条笔记”路径。
- 现有核心流程（新建、编辑、删除、文件夹切换、同步状态查看）无回归。

## 5. API 约定（当前与规划）

### 5.1 已实现（摘要）

- `POST /api/auth/login`：body `{ email, password }` → `{ token, user }`。
- `POST /api/upload/image`：multipart，返回 `{ storageId }`。
- `POST /api/notes/push`：multipart（`meta` + 图片二进制），幂等键 `clientNoteId`。
- `POST /api/snippets/upsert`：JSON，幂等键 `clientSnippetId`。
- `GET /api/files/:storageId`：读取已上传二进制流（JWT）。
- `POST /api/drive/folders/upsert`、`GET /api/drive/folders`。
- `POST /api/drive/files/push`、`GET /api/drive/files`、`GET /api/drive/files/:cloudId/download`、`DELETE /api/drive/files/:clientFileId`。

### 5.2 拉取相关（已实现摘要）

- `GET /api/notes`：返回当前用户笔记列表项（含 `clientNoteId`、`cloudId`、`updatedAt`、`images[]` 等，与 upsert 存储一致）。
- `GET /api/snippets`：返回当前用户短文本列表。
- Web：`pullFromCloud`（`@my-notes/sync-client`）下载图片写入 `blobs` / `images`，合并规则见 `@my-notes/shared`。
- **后续可扩展**：分页、`GET /api/notes/:id` 按需详情、冲突 UI。

## 6. 离线导出 / 导入

- **实现**：`apps/web/src/lib/export-archive.ts`；上传中心「导出 / 导入」Tab。
- **格式**：单文件 JSON：`exportVersion`、`exportedAt`、各表数组 + `blobs[].dataBase64`（单文件体积可能较大，大数据量后续可改为 zip 分块）。
- **导入策略**：当前为**按 id 合并写入**（`put`）；PRD 要求的「清空后全量覆盖」需在 UI 增加二次确认后再实现。
